# 常见问题

这篇文档集中整理 Claude Code CLI 部署、模型配置、Windows / WSL 分支选择中最容易遇到的问题。遇到报错时，先按对应问题逐项排查。

## 1. 输入 `claude` 提示找不到命令

按顺序检查：

1. 是否已经安装 Node.js。
2. 是否已经执行 `npm install -g @anthropic-ai/claude-code@latest`。
3. 安装后是否重新打开了终端。
4. 是否在 Windows 终端和 WSL 终端之间混用了安装位置。

如果你走的是 WSL 分支，请确认是在 WSL / Ubuntu 终端里执行 `claude`；如果你走的是 Windows 原生终端分支，请确认是在 Windows Terminal / PowerShell 里执行 `claude`。

## 2. 能启动 Claude Code，但无法回答

优先检查模型配置：

1. 如果使用 cc-switch，当前启用的配置是不是 Claude Code。
2. API Key 是否正确。
3. API 地址是否正确。
4. 模型名称是否存在。
5. 中转站余额或额度是否充足。
6. Windows / WSL 配置文件是否写到了当前正在使用的那套环境。

如果你不确定自己当前使用的是哪套配置，可以回到 [模型配置](./03-模型配置.md) 里的“先确认你要配置哪套环境”部分重新核对。

## 3. cc-switch 和手动配置可以混用吗？

不建议混用。

如果你使用 cc-switch，就尽量在 cc-switch 里改配置；如果你手动改了 `settings.json`，下一次 cc-switch 切换配置时可能会覆盖你的手动修改。

如果你已经混用过，建议选择一种方式重新整理：

- 想省心：回到 cc-switch，重新启用一次 Claude Code 配置。
- 想完全手动：关闭 cc-switch 的自动切换能力，只维护 `settings.json`。

## 4. Windows 10 终端显示错乱

优先使用 Windows Terminal，不建议继续使用老版本 `cmd`。

如果你还没有安装 Windows Terminal，请回到 [Windows 分支](./01-Windows分支.md) 查看 Windows 10 的终端安装步骤。

## 5. 什么时候应该用 WSL？

如果你只是跟着教程使用 Claude Code，优先选择 Windows 原生终端分支。

如果你符合以下情况，可以从一开始就选择 WSL 分支：

- 项目依赖强依赖 Linux 环境。
- Windows 原生终端中出现无法解决的兼容问题。
- 你已经熟悉 Linux 命令行。
- 你计划使用 omx、omc 等更偏 Linux/macOS 适配的开源 Agent 工具链。
- 你经常使用会自动调用 `grep`、`sed`、`awk`、`find` 等 Linux 命令的模型工具。

## 6. Windows 和 WSL 都装了 Claude Code，怎么判断当前用的是哪一个？

看你当前打开的终端类型：

| 当前终端 | 你正在使用的 Claude Code |
| --- | --- |
| Windows Terminal 的 PowerShell 标签页 | Windows 原生终端里的 Claude Code |
| Windows Terminal 的 Ubuntu / WSL 标签页 | WSL 里的 Claude Code |

也可以执行：

```bash
claude --version
```

如果命令能输出版本号，只能说明“当前这套环境”装了 Claude Code；它不能证明另一套环境也装好了。

## 7. WSL 里访问不到 Windows 本机的 Y3 MCP Server

优先检查 Windows 版本。

如果你的 Windows 11 版本低于 22H2，不建议走 WSL 分支，因为 WSL 镜像网络模式可能不可用，导致 WSL 里不能直接通过 `127.0.0.1` 访问 Windows 本机启动的 Y3 MCP Server。

如果你的系统满足 Windows 11 22H2 或更高版本，请回到 [WSL 分支](./02-WSL分支.md) 中的“配置 WSL 镜像网络”部分重新检查。
