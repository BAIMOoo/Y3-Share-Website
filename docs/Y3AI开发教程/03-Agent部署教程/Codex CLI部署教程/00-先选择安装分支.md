# 先选择安装分支

:::tip 验证与时效说明
AI 工具、模型名称、安装步骤和第三方服务变化很快。本文保留实践经验，但涉及时效性的内容，生产使用前请结合官方文档和来源记录再确认。
:::

Codex CLI 是 OpenAI 的终端 Agent，可以在本地项目目录里读代码、改代码、运行命令并解释报错。OpenAI 官方文档说明，Codex CLI 可以在终端中运行，并可读取、修改和运行当前目录中的代码。

开始安装前，请先在“Windows 原生终端分支”和“WSL 分支”之间做一次选择。两条路都能使用 Codex CLI，但适合的人不一样。

## 先用这张表做选择

| 你的情况 | 建议选择 | 原因 |
| --- | --- | --- |
| 第一次接触终端 Agent | Windows 分支 | 路径直观，文件都在 Windows 里，适合零基础用户 |
| 主要做 Y3 项目开发 | Windows 分支 | Y3 编辑器、项目文件、MCP Server 通常都在 Windows 侧 |
| Windows 10 用户 | Windows 分支 | 先安装 Windows Terminal，再按 Windows 分支操作 |
| Windows 11 版本低于 22H2 | Windows 分支 | 不建议走 WSL；WSL 镜像网络可能不可用，影响访问 Windows 本机的 Y3 MCP Server |
| 已经熟悉 Linux / WSL | WSL 分支 | Linux 工具链、脚本和开源 Agent 工具适配更自然 |
| 项目明确依赖 Linux 环境 | WSL 分支 | Linux 命令、依赖编译和脚本执行通常更顺手 |
| 后续想深入使用 omx、omc 等开源 Agent 工具链 | WSL 分支 | 这些工具通常更优先适配 Linux/macOS |

## 两条分支的具体差异

| 选择 | 优点 | 缺点 | 更适合谁 |
| --- | --- | --- | --- |
| Windows 分支 | 安装路径短<br/>项目文件都在 Windows 里<br/>和 Y3 编辑器、资源管理器配合直观<br/>Codex 官方也支持 Windows 原生 PowerShell | 个别开源脚本和命令行工具更偏 Linux<br/>遇到 Linux 风格命令时可能需要额外处理 | 大多数 Windows 10 / Windows 11 用户<br/>第一次接触 Agent 的用户<br/>主要做 Y3 项目开发的用户 |
| WSL 分支 | 更接近 Linux 开发环境<br/>很多开源教程可以直接套用<br/>适合复杂工程依赖<br/>omx、omc 等工具链更容易适配 | 多一层 Linux 子系统<br/>Windows 路径和 Linux 路径容易混淆<br/>Node.js、Codex、配置文件可能要维护两套 | 已经熟悉 Linux / WSL 的用户<br/>项目明确需要 Linux 环境的用户<br/>希望折腾更多开源 Agent 工具链的用户 |

## 我的建议

- 如果你是零基础，优先选择 <strong>Windows 分支</strong>。
- 如果你使用 Windows 10，先安装 Windows Terminal，再走 Windows 分支。
- 如果你使用 Windows 11，通常可以直接使用系统自带的 Windows Terminal。
- <strong>如果你的 Windows 11 版本低于 22H2，不建议选择 WSL 分支。</strong>WSL 的镜像网络模式需要 Windows 11 22H2 及以上；低于这个版本时，WSL 里可能无法通过 `127.0.0.1` 直接访问 Windows 本机启动的 Y3 MCP Server。
- 如果你已经熟悉 WSL，或者项目强依赖 Linux 环境，可以一开始就选择 <strong>WSL 分支</strong>。
- 两个分支可以同时安装，但 Windows 和 WSL 的配置文件是相互独立的，需要维护两份配置；零基础用户建议先选一条主线跑通。
