# 常见问题

这篇文档集中整理 Codex CLI 部署、模型配置、Windows / WSL 分支选择中最容易遇到的问题。

## 1. `codex` 命令找不到

按顺序检查：

1. Node.js 是否安装成功。
2. 是否执行过 `npm install -g @openai/codex@latest`。
3. 安装后是否重新打开终端。
4. 是否在 Windows 终端和 WSL 终端之间混用了安装位置。

如果你走的是 WSL 分支，请确认是在 WSL / Ubuntu 终端里执行 `codex`；如果你走的是 Windows 分支，请确认是在 Windows Terminal / PowerShell 里执行 `codex`。

## 2. 启动后 API 报错

优先检查模型配置：

1. 如果使用官方登录，是否已经按提示完成登录。
2. 如果使用 cc-switch，当前启用的是不是 Codex 配置。
3. API Key 是否复制完整。
4. API 地址是否带 `/v1`，以中转站文档为准。
5. 模型名称是否被中转站支持。
6. 账号余额或额度是否足够。

## 3. Windows 10 显示异常

请使用 Windows Terminal，不建议使用老版本 `cmd`。

如果你还没有安装 Windows Terminal，请回到 [Windows 分支](./01-Windows分支.md) 查看 Windows 10 的终端安装步骤。

## 4. 什么时候应该用 WSL？

零基础用户可以先尝试 Windows 分支。

如果你符合以下情况，可以从一开始就选择 WSL 分支：

- 项目依赖强依赖 Linux 环境。
- Windows 原生终端中出现无法解决的兼容问题。
- 你已经熟悉 Linux 命令行。
- 你计划使用 omx、omc 等更偏 Linux/macOS 适配的开源 Agent 工具链。
- 你经常使用会自动调用 `grep`、`sed`、`awk`、`find` 等 Linux 命令的模型工具。

## 5. Windows 和 WSL 都装了 Codex，怎么判断当前用的是哪一个？

看你当前打开的终端类型：

| 当前终端 | 你正在使用的 Codex |
| --- | --- |
| Windows Terminal 的 PowerShell 标签页 | Windows 里的 Codex |
| Windows Terminal 的 Ubuntu / WSL 标签页 | WSL 里的 Codex |

也可以执行：

```bash
codex --version
```

如果命令能输出版本号，只能说明“当前这套环境”装了 Codex；它不能证明另一套环境也装好了。

## 6. 可以直接让 Codex 修改项目吗？

可以，但建议第一次进入项目时先用 `hello` 或“请介绍当前项目，不要修改文件”这类安全请求测试。

确认模型和目录都正确后，再让 Codex 修改代码。重要项目建议先使用 Git 管理，避免误改后不好恢复。
